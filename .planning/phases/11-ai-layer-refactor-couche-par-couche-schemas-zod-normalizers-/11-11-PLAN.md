    ### AI Layer Structure (convex/_ai/)

    As of Phase 11, the Convex AI layer is organized as:

    - `convex/ai.ts` — thin action definitions (public API: `api.ai.*`). Each handler is `auth → buildPrompt → chat → schema.safeParse → return`.
    - `convex/_ai/providers.ts` — getProviders / getClient / getModel (multi-provider with fallback).
    - `convex/_ai/chat.ts` — safeParseJSON, withRetry, chatJSON, chatText.
    - `convex/_ai/auth.ts` — verifyAccessCode.
    - `convex/_ai/schemas.ts` — Zod schemas (single source of truth for AI output validation, lenient + passthrough for backward compat).
    - `convex/_ai/normalizers.ts` — `normalizeCVData` and friends (coerce raw LLM output into canonical CVData).
    - `convex/_ai/prompts/fragments.ts` — shared rule fragments (FABRICATION_GUARD, ACTION_VERBS_*, KPI_RULES_FR, INTRO_PRESERVATION_*).
    - `convex/_ai/prompts/*.ts` — one builder per action family: `extract.ts`, `adapt.ts` (shared by tailorCV + optimizeCVForPage), `rewrite.ts`, `analysis.ts`, `coverLetter.ts`.
    - `convex/_ai/__tests__/` — vitest specs for every pure helper (schemas, normalizers, each prompt builder).

    The underscore prefix keeps `_ai/` private: Convex treats only top-level files under `convex/` as API modules, so the sub-files never appear in the generated `_generated/api.d.ts`. The public surface (`api.ai.*`) stayed stable throughout the refactor.
